# AWQP Label Policy Draft

This draft translates the current AWQP Streamlit config and the legacy R dictionaries into a stricter policy for future labels plus a compatibility layer for old data.

The goal is to separate:

- canonical identifiers used for all new labels
- parent-child relationships between locations and treatments
- legacy aliases that should still be recognized by the R parser

## Core Policy

### 1. Canonical output format

All new labels generated by the Streamlit app should use:

`LOCATION-TREATMENT-EVENTNUMBER-EVENTTYPE-METHOD-ANALYTE[-D]`

Rules:

- `LOCATION` must always be present.
- `TREATMENT` may be blank only for locations that use no explicit treatment.
- A treatment must belong to one parent location.
- Treatment IDs must be globally unique and must not duplicate location IDs.
- Location labels and treatment labels should not duplicate each other in canonical data.

### 2. Parsing policy for old data

The R tool should remain tolerant of legacy IDs, but that tolerance should be explicit and separate from canonical naming.

Rules:

- Legacy aliases may map to a canonical location.
- Legacy aliases may map to a canonical treatment.
- Some legacy aliases may imply both a treatment and its parent location.
- Legacy aliases should not be emitted by the Streamlit label maker for new labels.

### 3. Data model policy

Each canonical treatment should carry:

- canonical treatment ID
- canonical treatment label
- parent location ID
- canonical aliases, if needed
- legacy-only aliases, if needed
- a flag for whether a legacy alias implies the parent location

Each canonical location should carry:

- canonical location ID
- canonical location label
- canonical aliases, if needed
- legacy-only aliases, if needed

## Proposed Legacy Alias Policy

Legacy aliases should be split into two kinds:

### A. Location aliases

These map directly to a canonical location and do not necessarily imply a treatment.

Examples:

- `A2` -> `ARDEC`
- `FW` -> `Fruita W`
- `FC` -> `Fruita C`
- `FF` -> `Fruita F`

### B. Treatment aliases that imply location

These map to a canonical treatment and should also infer the parent location when the location token is missing.

Examples:

- `AVST1` -> sample-code token `AVST1`, treatment group `ST`, location `AV`
- `FF3` -> treatment `F3`, location `FF`
- `SCA` -> treatment `SCA`, location `SC`
- `SB-SCO` -> treatment `SCO`, location `SC`

## Proposed Schema Shape For The Next Step

This is the structure I recommend for the YAML after you approve the policy.

```yaml
locations:
  SC:
    id: "SC"
    label: "Stagecoach"
    aliases: ["SC"]
    legacy_aliases: []

treatments:
  SCISC:
    id: "SCISC"
    label: "Stagecoach ISCO"
    parent_location: "SC"
    aliases: ["SCISC"]
    legacy_aliases: ["SC", "SB"]
    infer_parent_from_alias: true
```

For grouped analytical concepts such as Kerbel `ST` or `CT`, use a separate optional field:

```yaml
treatment_group: "ST"
```

That keeps parsing and label generation separate from analysis grouping.
