## Status

This policy is no longer just a draft concept. The following parts are already implemented in the app:

- canonical-only output for new labels

- location-scoped treatment choices in `Label Builder`
- legacy-only catalog entries for compatibility parsing
- live-generated ALS R dictionary export from the current catalog

## Core Policy

### 1. Canonical output format

All new labels generated by the app use:

`LOCATION-TREATMENT-EVENTNUMBER-EVENTTYPE-METHOD-ANALYTE[-D]`

Rules:

- `LOCATION` is always required.
- `TREATMENT` may be blank only for locations explicitly configured to allow blank treatment.
- Each non-blank treatment belongs to one parent location.
- Treatment IDs must be globally unique and must not duplicate location IDs.
- Canonical location labels and canonical treatment labels should not collide.
- Legacy aliases are for parsing compatibility only and should not be emitted for new labels.

### 2. Legacy parsing policy

Older IDs still need to be interpretable in R, but that logic is separate from canonical label generation.

Rules:

- A location may have `aliases` and `legacy_aliases`.
- A treatment may have `aliases` and `legacy_aliases`.
- A treatment alias may imply both the canonical treatment and its parent location.
- Legacy-only entries may remain in the catalog for compatibility, but should stay inactive for new label generation.

### 3. Implemented catalog schema

The current JSON catalog supports:

For locations:

- `id`
- `label`
- `aliases`
- `legacy_aliases`
- `allow_blank_treatment`
- `active`
- `legacy_only`

For treatments:

- `id`
- `label`
- `parent_location`
- `aliases`
- `legacy_aliases`
- `treatment_group`
- `r_label`
- `active`
- `legacy_only`

Special treatment:

- `blank` is the internal no-treatment option and has `id: ""`

## Label Builder Enforcement

The current `Label Builder` should enforce:

- only active locations are selectable
- once a location is selected, only treatments with that `parent_location` are shown
- `No treatment` only appears if the selected location allows blank treatment
- legacy-only and inactive entries do not appear as canonical builder options

This behavior is implemented in:

- [app.py](C:/Users/ansle/OneDrive/Documents/GitHub/awqp-label-maker/app.py)
- [utils/config_loader.py](C:/Users/ansle/OneDrive/Documents/GitHub/awqp-label-maker/utils/config_loader.py)

## Label Editor Policy

The `Label Editor` must preserve the parent-child model. That means:

- locations and treatments are still editable in separate tables
- but treatments must carry a valid `parent_location`
- active treatments cannot point to inactive parent locations
- deactivating a location with active child treatments should be blocked
- legacy-only entries may remain in the catalog without being exposed in canonical builder workflows

## ALS R Export Policy

The `ALS R Dicts` tab is generated live from the current catalog.

Rules:

- location output should include canonical location tokens plus child-treatment rescue tokens where needed for legacy parsing
- treatment output should respect `r_label` and `treatment_group` where applicable
- the R export is compatibility-oriented and may include legacy parsing support that is broader than the canonical builder output

## Resolved Decisions

These items were open earlier and are now settled in the current implementation:

- active catalog format is JSON, not YAML
- Stagecoach ISCO token is `SCISC`
- `Yampa 2` location token is `YTWO`

- `ARDEC` remains numeric-free for now
- compact Fruita labels remain preferred
